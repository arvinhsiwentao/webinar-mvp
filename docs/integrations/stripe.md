# Stripe（線上收款服務）

## 它是什麼

Stripe 是一個線上收款平台，幫我們在網站上收信用卡付款。你可以把它想像成「網路版的刷卡機」— 就像便利商店的收銀台有刷卡機一樣，Stripe 就是我們網站上的那台刷卡機。

本專案使用 Stripe 的「嵌入式結帳」（Embedded Checkout，意思是付款表單直接顯示在我們自己的網頁裡）。用戶不需要跳到 Stripe 的網站去付款，整個過程都在我們的網站上完成，體驗更順暢。

## 在專案中做什麼

Stripe 在我們的專案裡負責三件事：

1. **收錢** — 用戶在直播間看到商品，按下購買按鈕後，網頁上會出現一個付款表單，讓他們填信用卡資訊完成付款
2. **通知我們「錢收到了」** — Stripe 收到錢之後，會主動通知我們的伺服器，然後系統自動幫用戶領取啟用碼、寄確認信
3. **追蹤訂單** — 每筆訂單從建立到付款到完成，都有狀態紀錄，方便管理

## 需要設定的密碼和金鑰

Stripe 就像一個保險箱，需要好幾把鑰匙才能操作。這些「鑰匙」叫做環境變數（Environment Variables，存在伺服器上的機密設定值，程式啟動時會讀取）：

| 鑰匙名稱 | 做什麼用的 |
|------|------|
| `STRIPE_SECRET_KEY` | 伺服器專用的主鑰匙，用來跟 Stripe 溝通「幫我收錢」「幫我查訂單」等重要操作。絕對不能外洩 |
| `NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEY` | 給瀏覽器用的公開鑰匙，只能用來顯示付款表單，不能拿來收錢或查資料，所以公開沒關係 |
| `STRIPE_WEBHOOK_SECRET` | 用來驗證「真的是 Stripe 在跟我說話」的暗號。防止有人假冒 Stripe 騙我們說錢收到了 |
| `STRIPE_PRICE_ID` | 商品的價格編號，決定這次要收多少錢。在 Stripe 後台設定好價格後會拿到這個編號 |

### 用到的程式套件

| 套件名稱 | 做什麼用的 |
|------|------|
| `stripe` | 伺服器端的 Stripe 工具包，負責跟 Stripe 的伺服器溝通 |
| `@stripe/stripe-js` | 瀏覽器端的 Stripe 載入器，負責把付款表單帶到用戶的畫面上 |
| `@stripe/react-stripe-js` | React 專用的付款元件，讓我們可以把 Stripe 的付款表單像積木一樣嵌進網頁裡 |

## 付款的完整流程

想像你在便利商店買東西的過程，我們的付款流程類似：

```
【第一步：拿商品去結帳】
用戶在直播間或結束頁按下「立即購買」按鈕
  ↓
網頁跳轉到結帳頁面 /checkout/[webinarId]?email=xxx
  ↓

【第二步：收銀台準備刷卡機】
瀏覽器向我們的伺服器請求一組「結帳憑證」
  ↓
伺服器（POST /api/checkout/create-session）做以下事情：
  ├── 確認資料齊全（哪場研討會 + 用戶 email）
  ├── 查一下這場研討會存不存在
  ├── 檢查這個人是不是已經買過了 → 如果買過，告訴他「你已經購買過了」
  ├── 跟 Stripe 說：「請準備一個付款表單，金額是 XX 元」
  ├── 在資料庫記下這筆訂單（狀態：等待付款）
  └── 把「結帳憑證」傳回給瀏覽器
  ↓

【第三步：刷卡】
瀏覽器用這個憑證顯示 Stripe 的付款表單
用戶填入卡號 → 按下付款
  ↓

【第四步：確認收款 — 雙重保險機制】

  為什麼要「雙重保險」？因為網路有時候不穩定。
  就像你寄掛號信，既有郵差送信通知，又能上網查詢一樣，
  我們用兩條路線確保不會漏掉任何一筆成功的付款。

  路線 A — Stripe 主動通知（主要管道）：
    Stripe 收到錢後，會自動發一個通知
    （Webhook，像是快遞到貨時按你家門鈴）
    到我們的伺服器 → POST /api/checkout/webhook
    → 伺服器先驗證這個通知真的來自 Stripe（不是壞人偽造的）
    → 然後開始處理訂單

  路線 B — 我們主動去問（備用管道）：
    付款後瀏覽器會跳到確認頁面
    → 每 2 秒去問伺服器：「這筆訂單處理好了嗎？」
    → 如果發現 Stripe 說錢已到帳、但訂單還沒處理
    → 就直接在這裡處理（以防 Webhook 那條路線延遲或失敗）

【第五步：出貨 — fulfillOrder() 流程】

  不管是路線 A 還是路線 B 觸發，出貨流程都一樣：

  1. 上鎖：把訂單狀態從「等待付款」改成「已付款」
     └── 這裡有個巧妙的設計：只有第一個改成功的人算數
         （原子鎖，像搶票一樣，同一張票只能賣給一個人）
         這樣就算路線 A 和路線 B 同時到達，也不會重複出貨
  2. 從 Google Sheets 領取一組啟用碼
  3. 把啟用碼存進訂單紀錄，狀態更新為「已完成」
  4. 寄確認信給用戶（背景執行，不影響主流程）
  5. 記下一筆操作紀錄（留底備查）
```

### 訂單的三個階段

就像網購的物流狀態一樣，每筆訂單會經歷三個階段：

```
pending（等待付款）— 用戶開始結帳時建立
  ↓ 信用卡扣款成功
paid（已付款）— 錢收到了，開始處理
  ↓ 啟用碼已領取並寄出
fulfilled（已完成）— 一切搞定，啟用碼已交付
```

## 相關程式檔案

| 檔案位置 | 負責什麼 |
|------|------|
| `src/lib/stripe.ts` | Stripe 的初始化設定（像是把鑰匙插進刷卡機開機） |
| `src/lib/fulfillment.ts` | 出貨邏輯 — 上鎖防重複 + 領啟用碼 + 寄確認信，整個「收到錢之後該做的事」都在這裡 |
| `src/app/(public)/checkout/[webinarId]/page.tsx` | 結帳頁面（用戶看到付款表單的那一頁） |
| `src/app/(public)/checkout/[webinarId]/return/page.tsx` | 付款確認頁（付完款後跳到的頁面，會每 2 秒檢查訂單是否處理完） |
| `src/app/api/checkout/create-session/route.ts` | 建立結帳憑證的 API（伺服器跟 Stripe 說「準備收錢」） |
| `src/app/api/checkout/session-status/route.ts` | 查詢付款狀態的 API（「這筆錢到了嗎？」） |
| `src/app/api/checkout/webhook/route.ts` | 接收 Stripe 通知的 API（Stripe 按門鈴的入口） |

## 需要注意的重要事項

1. **雙重保險機制** — Webhook 通知 + 主動輪詢，兩條路都能觸發出貨。原子鎖保證不管哪條路先到，啟用碼只會發一次。這是為了確保：即使網路不穩，用戶付了錢一定能收到啟用碼

2. **付款表單嵌在網頁裡** — 用戶從頭到尾都在我們的網站上操作，不會被跳轉到 Stripe 的頁面。這樣體驗更好，用戶不會覺得「我被帶到一個奇怪的網站」而放棄付款

3. **防止重複購買** — 如果同一個 email 已經對同一場研討會下過單（不管是等待中、已付款、還是已完成），系統會回傳 409 錯誤（意思是「衝突，你已經買過了」），不讓他重複結帳

4. **訂單附帶資訊（Metadata）** — 每筆付款都會附上研討會編號、用戶 email、來源頁面等資訊。就像在匯款時備註「這是 XX 研討會的學費」一樣，方便日後查帳和對帳

5. **輪詢超時處理** — 付款確認頁會每 2 秒檢查一次，如果 30 秒後訂單還沒處理完，會顯示「訂單處理中，請稍候」的提示，讓用戶不會乾等著焦慮

6. **啟用碼領取失敗的容錯** — 如果從 Google Sheets 領取啟用碼時出了問題，訂單會退回「等待付款」狀態，等待下一次 Webhook 重試或輪詢重試。這樣確保不會出現「錢收了但沒給碼」的情況
