# SendGrid（自動寄信服務）

## 它是什麼

SendGrid 是一個「自動寄信機器人」—— 你給它收件人地址和信件內容，它就幫你把信寄出去。就像郵局一樣，你不用自己跑去寄，把信交給它就好。

本專案用 SendGrid 在幾個關鍵時刻自動寄信給用戶：報名成功時、直播快開始時、以及購買完成時。

## 為什麼用 fetch 而不是 SDK

SDK（軟體開發工具包，可以想成一個「大型工具箱」）是 SendGrid 官方提供的現成程式庫。但它體積大、附帶很多我們用不到的東西。

所以我們選擇用 `fetch`（一種直接發送網路請求的方式，像是自己寫信封貼郵票，而不是買整套文具組）。這樣只需要幾十行程式碼就能完成寄信，更輕量、更好維護。

## 在專案中做什麼

系統會在 4 個時機自動寄信：

| 信件類型 | 什麼時候寄 | 從哪裡觸發 |
|---------|---------|---------|
| 報名確認信 | 用戶填完報名表、按下送出後 | `POST /api/register`（報名處理程式） |
| 24 小時提醒信 | 直播開始前大約一天 | `GET /api/cron/reminders`（定時排程） |
| 1 小時提醒信 | 直播開始前大約一小時 | `GET /api/cron/reminders`（定時排程） |
| 購買確認信 | 付款成功之後 | `fulfillOrder()`（訂單履行程式） |

你可以把它想像成一個盡責的助理：報名時寄「歡迎信」、開播前寄「別忘了來看」、買完東西寄「收據和啟用碼」。

## 關鍵配置

### 環境變數（系統運行需要的密碼和設定值）

| 變數名稱 | 做什麼用 | 沒設定會怎樣 |
|---------|---------|---------|
| `SENDGRID_API_KEY` | SendGrid 的「通行證」，有它才能寄信 | 信寄不出去，但系統其他功能照常運作（只會在後台印一行警告） |
| `FROM_EMAIL` | 信件上顯示的寄件人 email | 自動用 `noreply@webinar.example.com` |
| `FROM_NAME` | 信件上顯示的寄件人名字 | 自動用 `Webinar` |
| `CRON_SECRET` | 定時提醒功能的「門禁密碼」 | 提醒信的排程會被擋下來（回傳 401 未授權） |

## 資料怎麼流動

用一個比喻來說明：想像你開了一家店，有個自動寄信的流程——

```
第 1 步：某件事發生了（有人報名 / 到了提醒時間 / 有人付款成功）
  ↓
第 2 步：系統呼叫「寄信功能」，把收件人、標題、內容準備好
  ↓
第 3 步：檢查有沒有 SendGrid 通行證（API Key）
  ├── 沒有通行證：在後台記一筆「信沒寄出」→ 結束（不影響其他功能）
  └── 有通行證：
      ↓
      第 4 步：把信交給 SendGrid（透過網路請求送到 SendGrid 的伺服器）
      ↓
      寄成功：在後台記一筆「信已寄出」
      寄失敗：在後台記一筆「寄信失敗」（但不會讓整個流程當掉）
```

### 最重要的設計原則：信寄不出去也沒關係

所有寄信動作都是「順便做」而非「必須做」—— 寄信失敗絕對不會拖垮主要功能：

- **報名時信沒寄出** → 用戶還是報名成功了，頁面正常跳轉
- **購買後信沒寄出** → 啟用碼已經存進資料庫了，網頁上照樣看得到
- **連 API Key 都沒設定** → 系統正常運行，只是後台會印一行提醒

為什麼這樣設計？因為「能不能寄信」和「核心業務能不能運作」是兩回事。不能因為郵局今天休息，就讓你的店也跟著關門。

## 信件模板（每封信長什麼樣）

所有信件都用 HTML 行內樣式（inline style）來排版。為什麼？因為大部分收信軟體（Gmail、Outlook 等）不支援外部 CSS 樣式表——你得把顏色、字體大小等設定直接寫在每個元素上，信才會長得正確。

### 報名確認信
- 頂部有一條金色漸層裝飾條（品牌視覺）
- 講師頭像 + 品牌名稱
- 活動時間（會根據用戶所在時區自動轉換，支援美東 ET / 美中 CT / 美西 PT / 台北 TST）
- 列出 5 項活動亮點（目前是寫死在程式碼裡的）
- 一個「進入直播間」的大按鈕（附帶追蹤參數，用來統計多少人從信裡點進來）

### 提醒信
- 24 小時提醒：「明天見！」
- 1 小時提醒：「建議提前 5 分鐘進入」
- 同樣附帶「進入直播間」按鈕

### 購買確認信
- 訂單資訊表格（日期、訂單號、商品名稱）
- **啟用碼**（用大字體 + 金色邊框特別突出，讓用戶一眼就看到）
- 5 步啟用教學（手把手教用戶怎麼用啟用碼）
- 3 個商品相關連結
- 客服聯絡方式

## 定時提醒怎麼運作

定時提醒像一個鬧鐘：外部的排程服務（cron service，可以想成「定時打電話叫你起床的服務」）每隔一段時間就呼叫我們的提醒端點。

具體流程：

1. **觸發**：外部排程服務發送請求到 `GET /api/cron/reminders`
2. **驗證身份**：請求必須附帶正確的密碼（`CRON_SECRET`），否則被拒絕
3. **查找需要提醒的人**：系統逐一檢查所有已發布的研討會 → 逐一檢查每位報名者 → 根據他們被分配的場次時間，算出離開播還有多久 → 決定該寄哪種提醒信
4. **注意：沒有防重複機制** —— 如果排程不小心多跑了一次，同一個人可能會收到重複的提醒信。目前是靠控制排程頻率來避免這個問題

## 注意事項

1. **不用 SDK** —— 直接用最基本的 `fetch` 呼叫 SendGrid v3 API（原因見上方「為什麼用 fetch」）
2. **只寄 HTML 格式** —— 不另外發純文字版。大多數用戶都用支援 HTML 的收信軟體，所以影響不大
3. **UTM 追蹤參數**（用來追蹤流量來源的標記）—— 所有信件裡的按鈕都會帶上 `utm_source=email`，加上用戶報名時原本的來源追蹤參數
4. **時區支援** —— 信件中的日期時間支援 4 個時區的中文格式化：美東、美中、美西、台北
5. **Follow-up 信尚未啟用** —— 程式碼裡已經寫好了 `followUpEmail()` 函數，但還沒有任何地方會呼叫它（預留給未來使用）
6. **部分內容是寫死的** —— 購買確認信裡的商品名稱、連結、到期日目前直接寫在程式碼裡，改動時需要去改程式碼
