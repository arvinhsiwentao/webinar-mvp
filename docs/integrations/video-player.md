# Video.js + HLS.js（影片播放器）

## 它是什麼

Video.js 是一個「萬能影片播放器」—— 你給它一個影片網址，它就能在網頁上播放。HLS.js 是它的搭檔，專門負責處理 HLS 格式（HTTP Live Streaming，一種把影片切成小片段、邊下載邊播放的技術，Netflix 和 YouTube 都是類似原理）。

兩個搭在一起，就能在瀏覽器裡流暢播放我們存在 Mux（影片託管平台）上的預錄影片。

## 在專案中做什麼

播放預錄影片，同時**讓觀眾以為自己在看真實直播**。

你可以把它想像成一個「電視台重播」的概念：影片是預先錄好的，但播出時看起來跟現場直播一模一樣。觀眾不知道影片是錄好的，就像你看電視重播時，如果沒有字幕寫「重播」，你可能也分不出來。

這就是本專案的核心 —— 所有播放器的設定、限制、視覺設計，全部都是為了維護這個「直播幻覺」。

## 怎麼製造「直播幻覺」

要讓預錄影片看起來像直播，我們做了很多手腳。以下每一項都有明確的理由：

| 做了什麼 | 怎麼做的 | 為什麼要這樣做 |
|---------|---------|--------------|
| 不顯示影片總時長 | 把時長顯示關掉，再用 CSS（控制網頁外觀的程式碼）強制隱藏 | 真正的直播不可能知道「總共會播多久」 |
| 不讓觀眾快轉 | 如果有人拖進度條到「還沒播到」的位置，自動拉回來 | 直播不能「跳到未來」—— 未來還沒發生 |
| 不讓用鍵盤快轉 | 攔截方向鍵和 Home/End 鍵 | 同上，封堵所有快轉的途徑 |
| 隱藏進度條 | 在直播模式下完全關閉進度條 | 真正的直播沒有進度條 |
| 不用原生全螢幕 | 自己做一個「假全螢幕」（把整個區塊放大到佔滿畫面） | 如果用瀏覽器內建的全螢幕，瀏覽器會自動顯示原生進度條，暴露影片總時長 |
| 不允許畫中畫 | 關閉「小窗播放」功能 | 小窗模式會顯示瀏覽器的播放控制項，也會暴露 |
| 顯示 LIVE 標籤 | 在播放器左下角注入紅色圓點 + "LIVE" 文字 | 視覺上強化「這是直播」的感受，就像電視台左上角的紅色 LIVE 標誌 |
| 只顯示「已播時間」 | 只顯示「從開播到現在過了多久」 | 像直播一樣只知道「開播了多長時間」，不知道還剩多久 |
| 不允許變速播放 | 關閉 1.5x、2x 等速度選項 | 直播不能加速 |

### 拖拉限制的原理（Wall-Clock 機制）

這個機制的意思是：觀眾最多只能拖到「直播開始到現在的實際時間」。

舉個例子：

```
假設直播排定在下午 2 點開始
現在是下午 2 點 25 分
→ 從開播到現在過了 25 分鐘 = 1500 秒

那麼觀眾最多只能拖到影片的 1500 秒位置
如果他嘗試拖到 1800 秒（30 分鐘處），系統會自動把他拉回 1500 秒
（有 0.5 秒的容差，避免正常播放時誤觸發限制）
```

為什麼？因為在真正的直播裡，你可以倒帶看剛才的片段，但不可能看到「還沒播出的內容」。

系統會把「開播時間」存在 `sessionStorage`（瀏覽器暫存區）裡，所以即使觀眾重新整理頁面，這個限制也不會重置。

## 自動播放怎麼處理

現代瀏覽器對「自動播放影片」有嚴格限制（為了防止網頁一打開就大聲播放影片嚇到人）。我們的處理方式像一個有備案的流程：

```
進入直播間，嘗試自動播放
  ├── 成功（有聲音）→ 太好了，正常播放 ✓
  ├── 成功（但被瀏覽器強制靜音）→ 顯示一個「點擊取消靜音 🔊」的提示
  └── 完全失敗 → 3 秒後顯示一個「點擊播放」的按鈕
```

另一個情況：觀眾切到其他分頁（比如去看 Gmail），再切回來時：

```
觀眾回到直播間
  ↓
自動跳到「現在應該播到的位置」（避免畫面停在剛才的地方，不像直播）
  ↓
嘗試恢復有聲播放
  ├── 成功 → 有聲音繼續看
  └── 失敗 → 先靜音繼續播（顯示取消靜音提示）
```

## 全螢幕怎麼做

**問題**：如果用瀏覽器內建的全螢幕功能，它會顯示原生的影片控制列 —— 包括進度條和影片總時長。這會直接戳破「直播幻覺」。

**解決方案**：我們全螢幕放大的是「包住影片的那個容器」，不是影片本身。就像在牆上掛了一台超大電視，但遙控器（控制項）還是我們自己做的那個。

```
方法 1（大多數瀏覽器）：
  把包住影片的容器放大到全螢幕 + 螢幕鎖定橫向

方法 2（舊版 iPhone 的備用方案）：
  用 CSS 把容器固定在畫面最前面，蓋住所有東西
  └── 同時攔截手機「返回」鍵，避免意外退出
```

## 遲到的觀眾怎麼處理（Late Join）

如果直播已經開始了 15 分鐘，觀眾才進來，系統會自動跳到正確的位置：

```
直播排定在下午 2:00 開始
觀眾在下午 2:15 才進來
→ 系統算出：已經播了 15 分鐘 = 900 秒
→ 自動把影片跳到 900 秒的位置開始播放
```

這就像你遲到了 15 分鐘走進電影院，但電影不會從頭放給你看 —— 你直接從現在的進度開始看，跟其他觀眾同步。

計算遲到時間的邏輯在直播間的頁面程式碼裡（`calculateLateJoinPosition()` 函數）。

## 觀看行為追蹤

播放器會把觀眾的觀看行為回報給 GA4（Google Analytics 4，一個網站分析工具），讓我們知道觀眾看了多少、看多久：

| 什麼時候回報 | 回報什麼 | 用白話文說 |
|------------|---------|----------|
| 影片每播一點點 | 觀看進度里程碑（5%, 10%, 15%…100%） | 「這個觀眾看到全片的 XX% 了」 |
| 每 60 秒 | 心跳信號 | 「這個觀眾還在看」（像定時打卡，追蹤實際觀看時長） |
| 影片播完 | 最後一次心跳 | 「這個觀眾看完了，總共看了 XX 分鐘」 |
| 影片播完 2 秒後 | 自動跳轉到結束頁面 | 影片結束後自動把觀眾帶到下一步（結束頁 → 可能的購買流程） |

## 需要設定的密碼和金鑰

無 —— 播放器本身不需要任何環境變數。影片的播放網址來自資料庫裡的研討會資料（由 Mux 提供）。

### 為什麼播放器要用特殊方式載入

播放器用 `next/dynamic`（Next.js 的動態載入功能）引入，並且關閉了伺服器端渲染（SSR）：

```typescript
const VideoPlayer = dynamic(() => import('@/components/video/VideoPlayer'), {
  ssr: false,
  loading: () => <LoadingSpinner />,
});
```

**為什麼**：Video.js 需要操作瀏覽器的畫面元素（DOM），而伺服器上沒有瀏覽器環境，所以必須等到程式碼跑在用戶的瀏覽器裡時才載入。就像你不能在沒有螢幕的電腦上「播放」影片一樣。

## 需要注意的重要事項

1. **禁止拖拉是商業需求，不是 Bug** —— 這是故意的。如果有人回報「觀眾不能快轉」，不要嘗試修復它。整個產品的商業模式就建立在「觀眾相信這是直播」的基礎上

2. **防重入鎖** —— 拖拉限制的程式碼裡有一個「鎖」（reentrancy guard），防止限制被重複觸發造成死循環。修改拖拉相關邏輯時，務必保留這個鎖

3. **0.5 秒的容差** —— 拖拉限制有 0.5 秒的緩衝。為什麼？因為正常播放時影片位置會有微小的跳動，如果限制太嚴格，會讓正常播放也被擋下來

4. **假全螢幕的返回鍵處理** —— 在舊版 iPhone 上，全螢幕是用 CSS 疊加實現的，我們用 `history.pushState`（瀏覽器的歷史記錄操作）來攔截返回鍵。如果你改了頁面的路由（網址跳轉）邏輯，要小心跟這裡衝突

5. **瀏覽器暫存的 key 名稱** —— 開播時間存在 `sessionStorage` 裡，key 的格式是 `webinar-wall-clock-{場次時間}-{影片網址}`。如果這個值被清掉，拖拉限制會重置（觀眾就能從頭看了）
