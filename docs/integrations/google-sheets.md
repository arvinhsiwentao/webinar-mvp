# Google Sheets API（啟用碼 + 訂單同步）

## 它是什麼

Google Sheets API（一種讓程式自動讀寫 Google 試算表的工具）讓我們的系統能「像機器人一樣」操作 Google 試算表。本專案用它做兩件事：

1. **領取啟用碼** — 用戶付款成功後，系統自動從試算表中「取走」一組啟用碼，像從倉庫取貨一樣
2. **訂單同步** — 每天定時把所有訂單資料「搬」到另一張試算表，讓業務團隊不用進後台就能看到銷售狀況

## 為什麼用 Google Sheets 而不是資料庫

啟用碼是由合作方（CMoney）預先生成的，他們把碼放在 Google Sheets 裡交給我們。用 Sheets 而不是資料庫（專門儲存資料的系統），原因很簡單：

- **非技術人員也能管理** — 業務同事可以直接在 Sheets 裡新增、查看碼庫，不需要學任何技術工具
- **不需要額外建表** — 省去在資料庫裡另外建結構的工作
- **碼已經在 Sheets 裡了** — 直接對接最省事，不需要先搬到別的地方

## 在專案中做什麼

### 功能 1：啟用碼領取（像從倉庫取貨）

想像試算表是一個倉庫，每一列是一個貨架格，裡面放著一組啟用碼。流程是這樣的：

```
用戶付款成功（透過 Stripe 通知或前端查詢）
  ↓
fulfillOrder()（完成訂單的函式）— 先用「原子鎖」確保同一筆訂單只處理一次
  ↓
claimActivationCode(orderId, email)（領碼函式）
  ↓
讀取整張試算表，找到第一個「還沒被拿走」的碼
  ↓
在那一列寫上：已使用、訂單編號、買家 Email、日期
  ↓
再讀一次同一列 → 確認是不是「我們」寫的（防止被別人搶走）
  ├── 確認是我們的 → 成功！把啟用碼回傳給用戶
  └── 發現被別人搶了 → 換下一個空碼重試（最多 3 次）
```

> **什麼是「原子鎖」？** 就像公共廁所的門鎖——第一個人轉了鎖，其他人就進不去。在資料庫裡，`fulfillOrder()` 會把訂單狀態從「待處理」改成「處理中」，只有第一個成功改掉的程序能繼續領碼，其他同時進來的請求會被擋住。

### 功能 2：訂單同步（像每天印一份新報表）

```
外部定時任務 → GET /api/cron/orders-sync（帶認證密碼）
  ↓
從資料庫撈出所有訂單
  ↓
把試算表「Orders」分頁的舊資料全部清掉（保留表頭）
  ↓
把最新的全部訂單寫進去
  ↓
回傳結果，例如 { synced: 42 }（代表同步了 42 筆）
```

> 每次都是「全部清掉重寫」，不是只新增變動的部分。好處是簡單不易出錯，壞處是資料量大時會比較慢。

## 關鍵配置

### 環境變數（程式啟動時需要的設定值）

| 變數 | 用途 | 缺少時會怎樣 |
|------|------|-------------|
| `GOOGLE_SERVICE_ACCOUNT_KEY` | Service Account（服務帳號）的 JSON 憑證，讓程式有權限讀寫 Sheets | 啟用碼領取**失敗**，訂單同步也**失敗**，訂單會回滾到「待處理」等下次重試 |
| `CRON_SECRET` | 定時任務端點的認證密碼，防止外人亂呼叫 | 同步請求會被拒絕（回傳 401 未授權） |

### 試算表 ID（寫死在程式碼 `src/lib/google-sheets.ts` 裡）

| 用途 | Spreadsheet ID | 讀寫範圍 |
|------|---------------|----------|
| 啟用碼倉庫 | `1W9tK97n...` | `A:E`（A 到 E 欄） |
| 訂單報表 | `1sba5HDJ...` | `Orders!A:M`（Orders 分頁，A 到 M 欄） |

> 如果要換一張試算表，需要直接改程式碼裡的 ID，目前沒有從設定檔讀取的機制。

## 資料結構

### 啟用碼試算表（每列就是一個「貨架格」）

| A 欄 | B 欄 | C 欄 | D 欄 | E 欄 |
|------|------|------|------|------|
| 啟用碼本身 | 是否已領取（TRUE 代表已用，空白代表還沒） | 訂單 ID | 買家 Email | 領取日期（yyyy/mm/dd） |

### 訂單同步試算表（13 欄完整訂單資訊）

表頭依序為：ID、Webinar ID、Email、Name、Status、Amount、Currency、Activation Code、Created At、Paid At、Fulfilled At、Product Package ID、Sales Code。

## 搶碼保護機制（競爭條件處理）

這是整個整合中**最關鍵的設計**。為什麼需要它？

想像兩個收銀員（Webhook 通知和前端查詢）同時收到「付款成功」的消息，兩個人可能同時去倉庫拿碼：

```
收銀員 A 看到第 5 格有碼 → 拿走，寫上自己的名字
收銀員 B 也看到第 5 格有碼 → 也拿走，把 A 的名字蓋掉
```

這樣一個碼就被發給兩個人了！

**解決辦法：拿完回頭確認**

```
收銀員 A 寫上名字後 → 回頭再看一次 → 名字還是自己的 → 確認成功 ✓
收銀員 B 寫上名字後 → 回頭再看一次 → 名字變成 A 的了（被覆蓋了）→ 去拿第 6 格
```

再加上前面提到的「原子鎖」（資料庫層面確保 `待處理 → 處理中` 只有一個人能成功），形成**雙重保護**，確保每筆訂單只會拿到一組碼。

## 相關程式碼檔案

| 檔案 | 說明 |
|------|------|
| `src/lib/google-sheets.ts` | 核心邏輯：`claimActivationCode()`（領碼）+ `syncOrdersToSheet()`（訂單同步）+ Google 認證 |
| `src/lib/fulfillment.ts` | `fulfillOrder()`（完成訂單）— 內含原子鎖，呼叫領碼函式，失敗時回滾 |

| `src/lib/cron-auth.ts` | 定時任務的密碼驗證邏輯（`CRON_SECRET` Bearer token） |

## 注意事項

1. **沒有備用方案** — 如果 `GOOGLE_SERVICE_ACCOUNT_KEY` 沒設定，領碼會直接失敗，訂單狀態會被回滾到「待處理」，等下一次 Webhook 重試或前端再次查詢時再嘗試。不會悄悄跳過，也不會用假碼代替
2. **試算表 ID 寫死在程式碼裡** — 要換試算表就得改程式碼（`src/lib/google-sheets.ts` 最上方的常數），沒有外部設定的方式
3. **每次全量同步** — 訂單同步是「清空再重寫」，不是「只新增變動」。目前訂單量小所以沒問題，量大時可能需要改成增量同步
4. **Service Account 需要分享權限** — 在 Google Sheets 介面中，必須把試算表「分享」給 Service Account 的 Email 地址，並給予「編輯者」權限，程式才能讀寫
5. **重試上限 3 次** — 如果連續 3 個碼都被其他程序搶走（機率極低），系統會拋出錯誤，訂單回滾等下次重試
